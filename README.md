# IPI — Internal Recruitment Platform

A role-based recruitment workspace for **managers**, **bidders**, **interviewers** and **brokers**.

- Developer profile pool with CV upload (stored in **Supabase Storage**)
- Shared color-coded scheduling calendar
- Interview process pipeline (intro → tech → final → onboard)
- Live analytics for bidders, interviewers and brokers
- Teammates directory with contact channels (managers only)

Built with **React + Vite + TypeScript + Tailwind** on the frontend and **Express + Supabase (Postgres + Storage)** on the backend, with our own JWT auth (bcrypt-hashed passwords) on top of a dedicated `users` table.

---

## Setup — Supabase (one-time)

1. **Create a project** at <https://supabase.com> (free tier is fine).
2. **Run the schema.** Open the Supabase dashboard → **SQL Editor** → New query → paste the contents of [`server/db/schema.sql`](server/db/schema.sql) → Run. This creates the five tables and enables RLS (only the server's service-role key can read/write).
3. **Get your keys.** In the dashboard → **Project Settings → API Keys**:
   - `Project URL` (under **Project Settings → API**) → `SUPABASE_URL`
   - **Secret Key** (formerly called `service_role key`) → `SUPABASE_SERVICE_ROLE_KEY`
     - **Keep this secret.** It bypasses RLS — never expose it to the browser.
     - Do **not** use the "Publishable Key" — that one cannot bypass RLS and the server will not work with it.
4. **Create `server/.env`** from the example:
   ```env
   PORT=4000
   JWT_SECRET=replace-with-something-long-and-random
   CLIENT_ORIGIN=http://localhost:5173
   SUPABASE_URL=https://YOUR-PROJECT-REF.supabase.co
   SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOi...
   SUPABASE_STORAGE_BUCKET=cvs
   ```
5. **Seed default users + storage bucket:**
   ```bash
   npm --prefix server run seed
   ```
   This is idempotent — running it again is a no-op.

## Deploy (Vercel frontend + Render backend)

The frontend is a static Vite SPA; the backend is a long-running Express server.
Host them separately.

### 1. Push to GitHub
Commit the repo and push to a GitHub repository.

### 2. Backend → Render
1. <https://render.com> → **New → Blueprint** → connect the repo. It reads `render.yaml`
   and creates a web service from the `server/` folder. (Or **New → Web Service**,
   Root Directory `server`, Build `npm install`, Start `npm start`.)
2. Set environment variables:
   - `SUPABASE_URL` — your Supabase project URL
   - `SUPABASE_SERVICE_ROLE_KEY` — the Supabase **Secret Key**
   - `SUPABASE_STORAGE_BUCKET` — `cvs`
   - `JWT_SECRET` — auto-generated by the blueprint (or set your own)
   - `CLIENT_ORIGIN` — your Vercel URL (set after step 3; comma-separated for multiple)
3. Deploy. Note the URL, e.g. `https://ipi-api.onrender.com`. Verify
   `https://ipi-api.onrender.com/api/health` returns `{"ok":true}`.
4. Seed once from your machine (point `.env` at the prod Supabase, then
   `npm --prefix server run seed`) — or just create users via the **Users & Access** page
   after first login.

### 3. Frontend → Vercel
1. <https://vercel.com> → **Add New → Project** → import the repo.
2. **Root Directory:** `client`. Framework preset auto-detects **Vite**
   (Build `npm run build`, Output `dist`). `client/vercel.json` handles SPA routing.
3. **Environment Variable:**
   - `VITE_API_URL` = your Render backend origin, e.g. `https://ipi-api.onrender.com`
     (no trailing `/api`).
4. Deploy. Open the Vercel URL.

### 4. Connect the two
Copy the Vercel URL into the backend's `CLIENT_ORIGIN` on Render and redeploy.
(`*.vercel.app` and `*.netlify.app` are already allowed by CORS, but setting your exact
domain is cleaner — and required for custom domains.)

> **Netlify instead of Vercel?** Same steps — Base directory `client`, Build `npm run build`,
> Publish `client/dist`. `client/public/_redirects` handles SPA routing, and set the same
> `VITE_API_URL` env var.

> **Render free tier** spins down after inactivity, so the first request after idle takes
> ~30–60s. Fine for internal use; upgrade the plan to keep it always-on.

## Access control

| Feature              | Manager | Bidder              | Interviewer                | Broker                   |
|----------------------|---------|---------------------|----------------------------|--------------------------|
| Dashboard            | yes     | yes                 | yes                        | —                        |
| Developer profiles   | full    | full                | view only                  | —                        |
| Calendar             | full    | add / edit / delete | view & mark-done own only  | —                        |
| Interview processes  | full    | full                | view all                   | view assigned only       |
| Analytics            | full    | full                | full                       | —                        |
| Teammates            | full    | names only          | names only                 | —                        |
| Users & Access       | full    | —                   | —                          | —                        |

Brokers see only the **Interview Processes** page, filtered to processes assigned to them.
Teammate contact channels (email, telegram, discord, whatsapp) are visible to **managers only**; bidders and interviewers see names and roles only — enforced **server-side**.

## Notes

- CV files (PDF/DOC/etc.) live in the Supabase Storage bucket configured by `SUPABASE_STORAGE_BUCKET` (default `cvs`). The seed script creates it as a private bucket; the server streams downloads through `/api/developers/:id/cv` with auth enforced.
- The `users` table is **separate** from `supabase.auth.users` — we use our own JWTs so role-based access is simple to reason about. Migrate to Supabase Auth later if you need OAuth providers or email magic links.
- RLS is enabled on all tables but no policies are defined; only the server's `service_role` key can read/write. Don't expose that key in the frontend.
